## 1. 项目背景与目标

### 1.1 背景

当前 AI 编程智能体（如 Claude Code CLI）能力强大但缺乏企业级管理能力：
- 缺乏统一的任务抽象和生命周期管理
- 缺乏资源隔离和多租户支持
- 缺乏可观测性、审计和人工干预机制
- 缺乏任务编排和复用能力

### 1.2 目标

构建一个 **通用的智能体任务基础设施**：

- **初期**：封装 Claude Code CLI，提供编程任务的托管执行环境
- **长期**：扩展支持更多智能体类型和业务场景（运维、数据分析、内容生成等）

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| 通用性 | 不与特定 Agent 实现（Claude Code）强绑定，抽象层可替换 |
| 渐进式 | 先基于 Claude Code CLI 快速落地，验证后逐步扩展 |
| 可扩展 | 支持未来接入其他 Agent 运行时（如 OpenAI Agent、自研 Agent） |

### 1.4 实现策略

| 阶段 | 目标 |
|------|------|
| Phase 1 | 基于 Claude Code CLI 封装，快速验证 |
| Phase 2 | 抽象 Agent 接口，支持多运行时 |
| Phase 3 | 扩展到非编程场景 |

## 3. 核心功能需求

### 3.1 任务定义（Task Definition）

用户在创建任务时，可配置以下内容：

| 配置项 | 说明 |
|--------|------|
| 任务目标 | 任务要达成的目标描述 |
| 任务描述 | 详细的任务上下文和约束 |
| 成功 Metric | 可量化的任务成功标准 |
| 工具集 | 任务可使用的工具列表 |
| Skills | 任务可调用的技能包 |
| Agents | 参与任务的智能体角色 |
| Hooks | 任务生命周期钩子（前置/后置/异常） |
| 检查策略 | 任务结果验证规则 |
| 资源限制 | Token 上限、运行时长、循环次数上限 |

**任务定义格式**：YAML DSL

### 3.2 沙箱环境（Sandbox Environment）

- 任务启动时自动创建隔离的沙箱环境（容器）
- 容器预装基础能力和工具
- 根据任务配置，仅激活声明的工具/Skills，未声明的进行屏蔽
- 任务结束后环境回收

### 3.3 任务执行（Task Execution）

- 任务执行引擎接收任务定义，在沙箱中启动 Agent
- 支持反馈循环：未达成目标或 Metric 检查不通过时，根据反馈调整策略重新执行
- 执行过程实时记录，支持日志流式输出

### 3.4 状态管理与持久化（State & Persistence）

**任务状态**：

```
          ┌──────────────────────────────────────────────┐
          │                                              ▼
Pending → Scheduled → Running ←────────→ Paused
                          │                   │
                          ▼                   │
                   WaitingApproval            │
                          │                   │
              ┌───────────┼───────────────────┤
              ▼           ▼                   ▼
         Succeeded     Failed            Cancelled
                          │
                          ▼
                      Retrying → Running
```

| 状态 | 说明 |
|------|------|
| Pending | 任务已创建，等待调度 |
| Scheduled | 已调度，等待执行 |
| Running | 任务执行中 |
| Paused | 任务暂停，等待人工干预 |
| WaitingApproval | 等待审批断点 |
| Succeeded | 任务成功完成 |
| Failed | 任务执行失败 |
| Cancelled | 任务已取消 |
| Retrying | 任务重试中 |

**持久化需求**：
- 任务状态实时存储，支持断点恢复
- 执行日志完整记录，用于审计和改进
- 任务配置版本化管理

### 3.5 任务编排（Task Orchestration）

支持两种编排模式：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| DAG 静态编排 | 预定义任务依赖关系，按 DAG 执行 | 流程固定的场景 |
| Agent 动态编排 | 由规划 Agent 在运行时动态决定下一步任务 | 不确定性高、需要探索的场景 |

**编排能力**：
- 任务间参数传递
- 并行执行
- 条件分支
- 失败重试策略

### 3.6 人工干预（Human Intervention）

| 干预类型 | 说明 |
|----------|------|
| 审批断点 | 任务在关键节点暂停，等待人工审批后继续 |
| 实时接管 | 任务运行中可随时暂停，人工注入指令/修正后继续 |
| 异常介入 | 任务失败/超时/异常时触发人工介入 |

### 3.7 任务输出（Task Output）

| 输出类型 | 说明 |
|----------|------|
| 结构化报告 | JSON/Markdown 格式的任务执行报告 |
| 文件产物 | 生成的代码、配置文件、文档等，推送至 Git/对象存储 |
| 事件通知 | 输出到消息队列/Webhook，供下游系统消费 |

### 3.8 管理面板（Management Console）

**MVP 范围**：

| 功能 | 说明 |
|------|------|
| 运维监控 | 任务状态一览、资源使用、异常告警 |
| 开发调试 | 任务配置、日志查看、手动触发、调试运行 |
| 任务管理 | 中断、重试、取消任务 |

**后续迭代**：
- 任务编排可视化
- 人工审批工作流
- 统计报表

## 4. 非功能需求

### 4.1 部署架构

| 组件 | 说明 |
|------|------|
| 控制面 | SaaS 部署，运行在 Kubernetes |
| 执行面 | Pod / Kata Container 隔离 |

### 4.2 技术栈

| 组件 | 选型 |
|------|------|
| 数据库 | OceanBase |
| 缓存/队列 | Redis |
| 消息中间件 | SofaMQ |
| 容器编排 | Kubernetes |

### 4.3 多租户

- 支持多租户隔离
- 资源配额管理（CPU、内存、并发任务数）
- 数据隔离（租户间不可见）

### 4.4 可扩展性

- 控制面无状态设计，支持水平扩展
- 执行面按需扩缩容

### 4.5 可观测性

- 任务执行日志
- 资源使用监控
- 异常告警
- 审计追踪
