## 1. 产品概述

### 1.1 产品定位

Agentic Coding Platform 是一个 **通用的智能体任务执行平台**。

**底层能力**（通用）：
- 任务托管与生命周期管理
- 沙箱环境隔离与能力管控
- 执行引擎与反馈循环
- 人工干预与审批机制
- 任务编排（静态 DAG / 动态编排）
- 输出管理与事件通知

**上层场景**（可扩展）：

| 场景 | 说明 | 当前阶段 |
|------|------|---------|
| 开发者编码 | AI 驱动的代码开发、测试、提交、验收 | ✓ 核心场景 |
| 运维自动化 | 故障诊断、配置变更、巡检任务 | 后续扩展 |
| 数据分析 | 报表生成、数据清洗、洞察分析 | 后续扩展 |
| 内容生成 | 文档编写、翻译、内容审核 | 后续扩展 |

### 1.2 核心价值

| 价值点 | 说明 |
|--------|------|
| 标准化 | 统一的任务定义格式，降低 Agent 使用门槛 |
| 可控性 | 资源限制、能力边界、人工干预，确保执行安全 |
| 可观测 | 完整的执行日志、状态追踪、审计能力 |
| 可复用 | 任务模板化，支持跨团队共享 |
| 可扩展 | 底层能力通用，支持多场景接入 |

### 1.3 产品形态

| 形态 | 说明 |
|------|------|
| API 服务 | RESTful API，供程序化调用 |
| 用户面板 | Web UI，面向终端用户的任务操作界面 |
| 管理面板 | Web UI，面向管理员的运维监控界面 |
| CLI 工具 | 命令行工具，供开发者本地调试 |

### 2.2 阶段一：任务定义

**场景描述**：

开发者在一体化研发工作台领取开发任务，通过自然语言完成任务定义，将需求转化为 Agent 可执行的标准化 Task。

**任务定义要素**：

| 要素 | 说明 | 对应功能模块 |
|------|------|-------------|
| 角色定位 | Agent 在任务中扮演的角色 | 4.1 任务模板管理 |
| 交付目标 | 任务需要达成的具体目标 | 4.1 任务模板管理 |
| 工具集 | Agent 可使用的工具列表 | 4.3 沙箱环境管理 |
| 执行策略 | 任务执行的方式和约束 | 4.2 任务执行引擎 |
| 验收标准 | 任务完成的判定条件 | 4.1 任务模板管理 |
| 成功指标 | 可量化的成功度量 | 4.1 任务模板管理 |

**环境准备**：

系统自动为任务创建隔离运行环境，完成 Agent 实例的启动、配置初始化与核心上下文加载。

### 2.3 阶段二：自主执行与反馈闭环

**执行流程**：

Agent 执行过程中，基于任务进度按需渐进式加载上下文，完成以下动作：

| 阶段 | Agent 动作 | 对应功能模块 |
|------|-----------|-------------|
| 编码 | 自主完成代码编写 | 4.2 任务执行引擎 |
| 测试 | 生成单元测试 | 4.2 任务执行引擎 |
| 检查 | 代码格式化与合规检查 | 4.2 任务执行引擎 |
| 提交 | 自动生成 commit message 并提交 | 4.6 外部集成 |

**CI/CD 集成**：

提交后自动触发 CI 流水线，执行：

| 步骤 | 说明 | 对应功能模块 |
|------|------|-------------|
| 自动化测试 | 运行单元测试、集成测试 | 4.6 外部集成 |
| 架构校验 | 检查架构一致性 | 4.6 外部集成 |
| 质量门禁 | 代码质量检查 | 4.6 外部集成 |
| 安全扫描 | 漏洞扫描 | 4.6 外部集成 |
| 自动部署 | 部署到目标环境 | 4.6 外部集成 |

**反馈闭环**：

| 步骤 | 说明 | 对应功能模块 |
|------|------|-------------|
| 状态获取 | 获取 CI/CD 执行结果与应用运行状态 | 4.6 外部集成 |
| 结果分析 | 基于执行反馈分析问题 | 4.2 任务执行引擎 |
| 迭代修复 | Agent 自主启动迭代修复流程 | 4.2 任务执行引擎 |
| 循环验证 | 直至所有验收指标全部达成 | 4.2 任务执行引擎 |

### 2.4 阶段三：交付验收

**交付流程**：

Agent 完成全流程执行后，自动生成标准化 Merge Request（MR），提交至审核者进行人工审核验收。

| 步骤 | 说明 | 对应功能模块 |
|------|------|-------------|
| 生成 MR | 自动创建 Merge Request | 4.6 外部集成 |
| 人工审核 | 审核者审查代码变更 | 4.5 人工干预机制 |
| 验收通过 | 批准合并 | 4.5 人工干预机制 |
| 合并代码 | 合并到目标分支 | 4.6 外部集成 |

### 2.5 阶段四：知识沉淀与治理

**知识沉淀机制**：

| 沉淀类型 | 说明 | 对应功能模块 |
|----------|------|-------------|
| 失败案例 | 执行失败的反例自动沉淀至知识库 | 4.7 知识沉淀 |
| 人工标注 | 审核过程中的正反例通过标记入库 | 4.7 知识沉淀 |
| 执行轨迹 | 任务全链路执行 Trace 完整记录 | 4.2 任务执行引擎 |

**技术债治理**：

| 任务 | 说明 | 对应功能模块 |

### 2.6 场景五：目标驱动的异步 Agent 编排

> **版本规划**：本场景为远期愿景（v2.0+），不在 MVP/v1.x 范围内。当前记录用于指导架构预留。

**场景描述**：

开发者输入一个高层目标（如 PRD 文档、"支持本地开发环境"等），系统自动完成从目标拆解到多任务并行执行的全流程。人类仅需在关键决策点和 Code Review 时介入，通过 IM 通知异步响应，Agent 在等待期间可继续推进其他无依赖的任务。

**核心流程**：

```
开发者输入目标/PRD
        ↓
  系统自动拆解为 Issues（含依赖图、工作量估算）
        ↓
  展示拆解结果 → 开发者确认或调整
        ↓
  ┌─── 无依赖 Issues → Agent 并行开始实现
  │         ↓
  │    遇到设计决策 / 边界不清
  │         ↓
  │    IM 推送到开发者（手机 / 电脑）
  │         ↓
  │    开发者回复决策
  │         ↓
  │    Agent 继续工作
  │
  ├─── 代码完成 → 自动创建 PR
  │         ↓
  │    IM 推送 Code Review 请求
  │         ↓
  │    开发者 Review（批注 / 要求修改）
  │         ↓
  │    Agent 根据 Review 修改 → 重新提交
  │
  └─── 依赖链解锁 → Agent 自动开始下游任务
```

**与当前架构的关系**：

本场景并非全新的架构设计，而是现有模块能力的组合升级：

| 现有模块 | 当前能力（MVP） | 场景升级 |
|---------|---------------|---------|
| 任务模板管理 | 人工创建模板、选择模板 | Agent 根据目标自动生成任务定义 |
| 调度引擎 | 单任务 FIFO 队列 | 依赖图感知的多任务并行调度 |
| 执行引擎 | 单容器执行 | 多容器并行执行，依赖链自动流转 |
| 人工干预 | Dashboard 手动操作 | IM 异步通知 + 移动端响应 |
| 知识沉淀 | 后置记录 | 执行过程中实时积累，反哺后续任务 |

**产品价值**：

将开发者从"逐个任务的执行者"彻底解放为"目标定义者和质量把关者"。Agent 利用并行能力同时推进多个独立任务，人类只需在决策点和 Review 时异步介入，大幅提升开发吞吐量。

### 2.7 开发者角色转型

| 维度 | 传统模式 | Agent 驱动模式 |
|------|---------|---------------|
| 核心工作 | 编写代码 | 定义任务、审核验收 |
| 技能要求 | 编程能力 | 任务拆解、AI 协作 |
| 交付方式 | 逐行编码 | 自然语言描述目标 |
| 质量保证 | 人工 Review | Agent 自检 + 人工验收 |
| 知识积累 | 个人经验 | 系统化知识库 |

### 3.2 用户故事

#### 3.2.1 开发者

**US-D01：对话框式创建任务** `[场景追溯: 2.2 任务定义]` `[发布版本: v1.1]`

> 作为开发者，我希望通过对话框与系统交互，描述任务目标并回答系统追问，最终确认任务详情后提交，以便快速创建任务而无需填写复杂表单。

**验收标准**：
- 提供对话框界面，支持自然语言输入
- 系统自动识别意图并匹配任务模板
- 当信息不足时，系统主动追问引导用户补充
- 最终生成固定格式的任务详情页面供用户确认
- 用户确认后提交任务，无需查看底层配置

---

**US-D01a：基于模板创建任务** `[场景追溯: 2.2 任务定义]` `[发布版本: MVP]`

> 作为开发者，我希望通过选择任务模板并填写参数来创建任务，以便快速启动标准化的编码任务。

**验收标准**：
- 可浏览和搜索可用模板
- 选择模板后显示参数配置界面
- 支持填写模板定义的必填和可选参数
- 参数校验通过后可预览任务配置
- 确认后提交任务

---

**US-D01b：任务调试** `[场景追溯: 2.3 自主执行]` `[发布版本: MVP]`

> 作为开发者，我希望在任务执行过程中进行调试，包括查看中间状态、注入测试指令，以便验证任务模板或排查问题。

**验收标准**：
- 支持查看任务执行的中间状态和上下文
- 支持注入测试指令验证执行逻辑
- 支持单步执行（可选）
- 调试操作记录到执行日志

---

**US-D02：任务执行监控** `[场景追溯: 2.3 自主执行]`

> 作为开发者，我希望实时查看任务执行进度和日志，以便了解 Agent 的工作状态和中间结果。

**验收标准**：
- 实时显示执行进度和当前阶段
- 流式输出执行日志
- 显示资源消耗（Token、时间）
- 支持历史执行记录查看

---

**US-D03：人工干预任务** `[场景追溯: 2.3 反馈闭环]`

> 作为开发者，我希望在任务执行过程中进行人工干预，以便纠正 Agent 的错误方向或提供额外指导。

**验收标准**：
- 支持暂停/恢复/取消任务
- 支持注入指令继续执行
- 支持修改任务参数
- 记录干预历史

---

**US-D04：MR 审核验收** `[场景追溯: 2.4 交付验收]`

> 作为开发者（审核者），我希望对 Agent 生成的 MR 进行审核，以便确保代码质量后合并。

**验收标准**：
- 查看待审核的 MR 列表
- 查看 MR 详情和代码变更
- 支持批准/拒绝操作
- 支持添加审核意见

---

**US-D05：知识标注** `[场景追溯: 2.5 知识沉淀]` `[发布版本: v1.2]`

> 作为开发者，我希望在审核过程中对正反例进行标注，以便沉淀知识供后续任务参考。

**验收标准**：
- 支持标记代码片段为正例/反例
- 支持添加标注说明
- 标注自动关联到任务上下文
- 可查看历史标注记录

---

**US-D06：Provider 选择与配置** `[发布版本: MVP]`

> 作为开发者，我希望在创建任务时选择不同的 Agent 运行时（Provider），以便使用不同的模型（如 Claude、智谱 GLM 等）执行任务。

**验收标准**：
- 创建任务时可下拉选择 Provider（包含系统预置和租户/个人配置）
- 可配置个人 Provider（API 端点、密钥、模型映射等）
- 可设置个人默认 Provider
- Provider 配置兼容 cc switch 格式
- 可测试个人 Provider 连接

**Provider 选择优先级**：`任务指定 > 用户默认 > 租户默认 > 系统默认`

#### 3.2.2 平台管理员

**US-A01：租户管理**

> 作为平台管理员，我希望配置租户的资源配额和权限策略，以便确保资源合理分配和安全隔离。

**验收标准**：

## 4. 功能模块设计

### 4.1 任务模板管理

任务模板采用声明式配置，全面描述任务的定义、执行策略、生命周期管理和输出规范。模板设计遵循「定义即规范」原则，一份完整的模板应能够独立驱动任务的完整生命周期。

#### 4.1.1 模板整体结构

模板由以下五大模块组成：

| 模块 | 说明 | 必填 |
|------|------|------|
| **元数据（Metadata）** | 模板的基本信息、版本、标签 | ✓ |
| **任务定义（Spec）** | 任务目标、上下文、参数定义 | ✓ |
| **执行配置（Execution）** | 执行策略、资源限制、能力配置 | ✓ |
| **生命周期管理（Lifecycle）** | 状态转换、钩子、干预点、重试策略 | |
| **输出规范（Outputs）** | 输出类型、目标位置、通知配置 | |

#### 4.1.2 元数据（Metadata）

元数据定义模板的基本信息，包括：
- **标识信息**：名称、版本号、描述
- **分类信息**：场景标签（编码/运维/分析/内容）
- **生命周期**：状态（草稿/已发布/已废弃）、创建/更新时间

#### 4.1.3 任务定义（Spec）

任务定义描述任务要完成的目标和上下文：

- **任务目标**：任务要达成的目标，支持自然语言描述
- **任务描述**：详细的上下文、背景、约束条件
- **参数定义**：用户创建任务时需要填写的参数（支持多种类型和校验规则）
- **成功指标**：任务完成的判定条件（如输出检查、外部验证、超时等）
- **上下文配置**：代码仓库、知识库、环境变量、初始上下文

#### 4.1.4 执行配置（Execution）

执行配置定义任务的执行方式和资源约束：

- **能力配置**：Agent 可使用的工具集、Skills、网络策略、文件访问权限
- **资源限制**：Token 上限、执行时长、CPU/内存配额、并发限制
- **执行策略**：执行模式（同步/异步/定时）、优先级、超时行为、重试策略

#### 4.1.5 生命周期管理（Lifecycle）

生命周期管理定义任务的状态流转和干预机制：

- **状态配置**：初始状态、终止状态、状态转换规则
- **生命周期钩子**：在任务生命周期的关键节点执行的回调（如 onInit、onStart、onComplete、onSuccess、onFailure 等）
- **人工干预点**：定义需要人工介入的节点（审批/输入/审核）
- **重试策略**：失败后的重试次数、间隔、退避策略

#### 4.1.6 输出规范（Outputs）

输出规范定义任务完成后的产物和通知：

- **输出类型**：报告（Markdown/JSON）、文件产物、事件通知、执行指标
- **输出配置**：输出格式、内容模板、目标位置（Git/S3/Webhook）、触发条件
- **外部集成**：Git 仓库、CI/CD 流水线、知识库、通知渠道的集成配置

> 模板配置的详细字段定义、YAML Schema 规格参见 [TRD 第4章 控制面服务设计](./plans/2026-03-22-mvp-trd.md#4-控制面服务设计)。

#### 4.1.7 模板参数化

模板支持多层参数化：

| 层级 | 说明 |
|------|------|
| 模板参数 | 用户创建任务时填写的参数 |
| 环境变量 | 从运行环境注入的变量 |
| 系统变量 | 系统自动提供的变量（如任务ID、时间戳） |
| 上下文变量 | 从任务上下文提取的变量 |

**参数优先级**（从高到低）：

1. **上下文变量**：运行时从任务上下文动态提取，优先级最高
2. **模板参数**：用户创建任务时显式填写，覆盖默认值
3. **环境变量**：从运行环境注入，可作为模板参数的默认来源
4. **系统变量**：系统自动生成，不可被覆盖

**覆盖规则**：当同名参数存在于多个层级时，优先级高的覆盖优先级低的。敏感参数（如密码）不参与覆盖，仅从模板参数获取。

#### 4.1.8 场景化模板示例

| 场景 | 模板示例 | 核心特点 |
|------|---------|---------|
| 编码 - Bug 修复 | bug-fix | CI/CD 集成、MR 审核、代码上下文 |
| 编码 - 功能开发 | feature-dev | 完整生命周期、测试验证、文档生成 |
| 编码 - 代码审查 | code-review | 输出报告、知识沉淀 |
| 运维 - 故障诊断 | fault-diagnosis | 监控集成、告警联动 |
| 运维 - 配置变更 | config-change | 审批断点、回滚机制 |
| 分析 - 报表生成 | report-gen | 数据源集成、定时执行 |
| 内容 - 文档编写 | doc-writing | 模板输出、多格式支持 |

### 4.2 任务执行引擎

#### 4.2.1 执行流程

```
Create → Validate → Schedule → Start → Execute → Check → Complete
                                                      ↓
                                                  Reflect → Adjust → Retry
```

| 阶段 | 说明 |
|------|------|
| Create | 创建任务实例，加载模板配置 |
| Validate | 校验参数、权限、资源配额 |
| Schedule | 调度到执行队列 |
| Start | 创建沙箱环境，初始化能力，加载上下文 |
| Execute | Agent 在沙箱中执行任务 |
| Check | 检查成功指标是否达成 |
| Reflect | 分析执行结果，识别问题 |
| Adjust | 调整执行策略 |
| Retry | 重新执行（如需要） |

#### 4.2.2 Agent 反馈循环

Agent 执行采用 **目标-规划-执行-反思** 循环：

1. **Goal**：明确任务目标
2. **Plan**：制定执行计划
3. **Execute**：按计划执行动作
4. **Check**：检查是否达成目标
5. **Reflect**：反思执行结果
6. **Adjust**：调整策略
7. 循环直到成功或达到重试上限

#### 4.2.3 上下文管理

| 上下文类型 | 说明 |
|-----------|------|
| 任务上下文 | 任务目标、约束、验收标准 |
| 代码上下文 | 仓库结构、相关代码、依赖关系 |
| 知识上下文 | 相关历史案例、最佳实践 |
| 环境上下文 | 工具配置、环境变量 |

**渐进式加载**：
- 任务启动时加载核心上下文
- 执行过程中按需加载补充上下文
- 支持上下文缓存和复用

#### 4.2.4 执行轨迹记录

| 记录类型 | 说明 |
|---------|------|
| 动作日志 | Agent 执行的每个动作 |
| 决策点 | 关键决策及其依据 |
| 中间结果 | 各阶段的产出物 |
| 反馈记录 | 外部系统的反馈 |
| 资源消耗 | Token、时间、资源使用 |

#### 4.3.1 环境生命周期

| 阶段 | 动作 | 说明 |

### 4.4 任务编排引擎

#### 4.4.1 DAG 静态编排

**适用场景**：流程固定的确定性任务

**编排能力**：
- 任务依赖定义（A → B → C）
- 并行执行（A → [B, C] → D）
- 条件分支（基于上游任务结果）
- 失败重试策略
- 人工审批节点

**示例工作流**：

```
checkout ─────┬───→ lint ───┐
              │             │
              └───→ test ───┴───→ review ───→ deploy
```

#### 4.4.2 Agent 动态编排

**适用场景**：不确定性高、需要探索的任务

**工作原理**：
1. 规划 Agent 分析目标，制定子任务计划
2. 按计划创建并执行子任务
3. 评估执行进度
4. 根据评估结果动态调整计划
5. 循环直到目标达成或终止条件满足

**约束机制**：
- 最大子任务数量
- 最大编排深度
- 允许使用的模板范围
- 终止条件定义

| 类型 | 来源 | 说明 |

#### 4.7.2 知识入库机制

| 机制 | 触发条件 | 说明 |
|------|---------|------|
| 自动入库 | 任务失败 | 任务状态为 Failed 时，自动提取错误信息、执行轨迹、失败原因入库 |
| 自动入库 | 任务成功 | 任务状态为 Succeeded 且执行指标达标（如 Token 消耗低于阈值、循环次数少）时，提取成功模式入库 |
| 人工标注 | 审核时 | 用户在 MR 审核或任务详情页主动标记代码片段为正例/反例 |
| 关联入库 | 任务完成 | 任务完成后，执行轨迹与任务关联存储，用于后续分析 |

**自动入库判断条件**：

| 条件 | 失败案例入库 | 成功案例入库 |
|------|-------------|-------------|
| 任务状态 | Failed | Succeeded |
| 执行时长 | 任意 | > 30s（排除过短任务） |
| Token 消耗 | 任意 | < 模板配置上限的 80% |
| 循环次数 | 任意 | < 配置上限的 50% |
| 人工干预 | 任意 | 无人工干预（纯自主完成） |

### 4.11 Provider 管理

Provider 是 Agent 运行时的配置抽象，允许平台支持多种模型后端（如 Claude Code、智谱 GLM、DeepSeek 等），实现类似 `cc switch` 的模型切换能力。

#### 4.11.1 Provider 三层作用域

| scope | 说明 | 配置者 | 可见范围 |
|-------|------|--------|---------|
| **system** | 系统预置 Provider | admin | 所有用户 |
| **tenant** | 租户级 Provider | admin | 该租户所有用户 |
| **user** | 用户个人 Provider | developer | 仅该用户 |

#### 4.11.2 Provider 配置内容

Provider 配置兼容 `cc switch` 格式，包含以下核心配置：

| 配置类别 | 配置项 | 说明 |
|---------|--------|------|
| **API 配置** | api_endpoint | API 端点地址（如 Anthropic 兼容 API） |
| | api_key | API 密钥（通过 K8s Secret 安全存储） |
| **模型映射** | default_model | 默认使用的模型 |
| | opus_model | Opus 级别模型映射（如 glm-5） |
| | sonnet_model | Sonnet 级别模型映射（如 glm-4.7） |
| | haiku_model | Haiku 级别模型映射（如 glm-4.5-air） |
| **运行时配置** | runtime_type | cli / api / sdk |
| | runtime_image | Docker 镜像 |
| **环境变量** | env_vars | 额外环境变量注入 |
| **权限配置** | permissions | 允许/禁止的操作 |
| **插件配置** | enabled_plugins | 启用的 MCP 插件 |

#### 4.11.3 Provider 选择优先级

任务执行时的 Provider 选择链：

```
任务创建时指定 provider_id?
    │
    ├── 是 ──▶ 使用指定的 Provider
    │
    └── 否 ──▶ 用户有默认 Provider?
                    │
                    ├── 是 ──▶ 使用用户默认 Provider
                    │
                    └── 否 ──▶ 租户有默认 Provider?
                                    │
                                    ├── 是 ──▶ 使用租户默认 Provider
                                    │
                                    └── 否 ──▶ 使用系统默认 Provider (Claude Code)
```

**优先级**：`任务指定 > 用户默认 > 租户默认 > 系统默认`

### 5.3 任务状态流转

任务在其生命周期中会经历多种状态转换：

```
          ┌──────────────────────────────────────────────┐
          │                                              ▼
Pending → Scheduled → Running ←────────→ Paused
                          │                   │
                          ▼                   │
                   WaitingApproval            │
                          │                   │
              ┌───────────┼───────────────────┤
              ▼           ▼                   ▼
         Succeeded     Failed            Cancelled
                          │
                          ▼
                      Retrying → Running
```

**状态说明**：
- **Pending**：任务已创建，等待调度
- **Scheduled**：已调度，等待资源执行
- **Running**：执行中
- **Paused**：已暂停（人工干预）
- **WaitingApproval**：等待人工审批
- **Succeeded**：成功完成
- **Failed**：执行失败
- **Cancelled**：已取消
- **Retrying**：重试中

> 详细的状态转换条件、错误处理逻辑参见 [TRD 第6章 数据模型设计](./plans/2026-03-22-mvp-trd.md#6-数据模型设计)。

| MR | Merge Request，合并请求 |
| CI/CD | 持续集成/持续部署 |
| Trace | 执行轨迹，任务全链路执行记录 |
| Context | 上下文，任务执行所需的信息环境 |
| Knowledge | 知识库，沉淀的案例和经验 |
| Tech Debt | 技术债，代码中需要优化的部分 |
| Capability | 能力，工具/Skills/Agent 的统称 |
| Provider | Agent 运行时配置，定义使用的模型后端（如 Claude Code、智谱 GLM 等） |
| cc switch | Claude Code 的模型切换配置格式 |
