### 🟡 技术问题

#### 问题 3：cli-runner 启动脚本中的信号处理不完整

**位置**：5.4.1 cli-runner 启动脚本

**问题**：脚本使用 `set -e`，但没有捕获 SIGTERM 信号进行优雅关闭。当 K8s 发送 SIGTERM 时，脚本可能立即退出而不保存状态。

**建议修改**：

```bash
#!/bin/bash
set -e

# 添加信号处理
cleanup() {
    echo "Received termination signal, saving state..."
    echo "{\"status\": \"interrupted\", \"exit_code\": 130}" > /workspace/.agent-state/status.json
    exit 130
}
trap cleanup SIGTERM SIGINT

# ... 其余脚本内容
```

**严重程度**：🟡 中等

#### 问题 4：inject.json 读取机制未明确

**位置**：5.5.1 共享 Volume 通信

**问题**：文档提到 CLI 通过 inotify 监听 inject.json 或通过 wrapper HTTP 接口获取，但未明确最终采用哪种方案。这可能导致实现时的困惑。

**建议**：明确选择一种方案，并在 TRD 中详细说明实现方式。

**严重程度**：🟡 中等

## 四、设计合理性审查

### ✅ 合理的设计决策

| 决策 | 评价 | 说明 |
|------|------|------|
| 复用 Claude Code CLI 而非直接调用 API | ✅ | 降低开发成本，复用现有工具链 |
| Redis 优先级队列 + 自研调度器 | ✅ | 支持限流、抢占、排队可视化 |
| 敏感配置使用 K8s Secret | ✅ | 安全最佳实践 |
| 软删除支持 | ✅ | 数据安全，支持恢复 |
| API Key 使用 SHA256 哈希存储 | ✅ | 安全最佳实践 |

### 🟢 改进建议

#### 建议 1：考虑添加任务优先级配置的验证

**位置**：4.2 调度器

**建议**：虽然支持 high/normal/low 三种优先级，但建议添加：
- 租户级别是否允许使用 high 优先级的配置
- 高优先级任务的配额限制

#### 建议 2：考虑添加 Pod 资源配额的动态调整

**位置**：5.2.3 Job 规格配置

**建议**：当前 CPU/内存配额是静态配置的，后续可考虑根据任务模板动态调整。

#### 建议 3：考虑添加沙箱网络访问日志

**位置**：10. 安全设计

**建议**：建议记录沙箱的所有出站网络请求，用于安全审计和问题排查。

#### 建议 4：providers 表的 api_key_ref 字段安全性考虑

**位置**：6.2.9 providers 表

**建议**：对于 user 作用域的 Provider，用户自定义的 API Key 如何安全存储需要补充说明。建议：
- user 作用域的 API Key 仅存储在 K8s Secret 中
- Secret 名称与用户 ID 绑定
- 用户只能通过 API 创建/更新，无法直接读取
