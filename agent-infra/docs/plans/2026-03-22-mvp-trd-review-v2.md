## 2. 架构设计评价 ✅

### 2.1 Service 层设计（合理）

TRD 定义的 7 个 Service 模块职责清晰，边界合理：

| Service | 职责 | 评价 |
|---------|------|------|
| TaskSvc | 任务业务逻辑、状态管理 | ✅ 核心域 |
| TemplateSvc | 模板 CRUD、版本控制 | ✅ 核心域 |
| TenantSvc | 租户管理、配额控制 | ✅ 核心域 |
| UserSvc | 用户管理、API Key | ✅ 核心域 |
| CapabilitySvc | 工具/Skills 注册 | ✅ 核心域 |
| AuditSvc | 审计日志 | ✅ 横切关注点 |
| NotifySvc | 通知推送、WebSocket | ✅ 横切关注点 |

**结论**：服务划分遵循单一职责原则，为后续微服务拆分预留了良好边界。

### 2.2 沙箱 Pod 设计（合理）

3 容器模式符合"一个容器一个进程"的最佳实践：

| 容器 | 类型 | 职责 |
|------|------|------|
| cli-runner | 第三方进程 | Claude Code CLI 执行 |
| wrapper | 自研代码 | 监控、干预、状态上报 |
| log-agent | 开源组件 | 日志采集到 SLS |

**结论**：职责分离清晰，符合容器化最佳实践。

## 4. 文档臃肿问题 🟡

### 4.1 Dockerfile 内容过多

**位置**：TRD 11.2

**问题描述**：
包含 4 个完整的多阶段构建 Dockerfile（约 150 行）。

**分析**：
- Dockerfile 是实现细节，会在代码库中演进
- 设计文档中的完整代码容易与实际代码不同步
- 增加了文档维护负担

**建议**：
用规格表替代完整代码，保留关键构建要求：

```markdown
### 11.2 容器镜像规格

| 镜像 | 基础镜像 | 大小目标 | 安全要求 |
|------|---------|---------|---------|
| control-plane | alpine:3.19 | <30MB | 非 root, healthcheck |
| frontend | nginx:alpine | <50MB | 仅静态文件 |
| cli-runner | ubuntu:22.04 | <500MB | 含 Claude CLI |
| agent-wrapper | alpine:3.19 | <20MB | 非 root, healthcheck |

### 11.2.1 构建要求
- 所有镜像使用多阶段构建
- 所有镜像以非 root 用户运行 (UID 1000)
- 所有镜像包含健康检查
- CI 中扫描镜像漏洞

[实际 Dockerfile: deploy/dockerfiles/]
```

### 4.2 Makefile 内容过多

**位置**：TRD 11.3

**问题描述**：
完整的 100+ 行 Makefile 代码。

**建议**：
精简为命令规格表，保留实际 Makefile 链接。

### 4.3 CI/CD 配置内容过多

**位置**：TRD 11.4

**问题描述**：
完整的 GitLab CI YAML 配置。

**建议**：
用流程描述替代完整 YAML，保留关键阶段说明。

## 5. 缺少的规范 💡

### 5.1 状态转换矩阵

**位置**：TRD 6.2.5 tasks 表

**问题描述**：
定义了 9 个任务状态，但缺少合法的状态转换规则。

**建议补充**：

```markdown
### 6.3 任务状态转换规则

| 从状态 | 可转换到 | 触发条件 |
|--------|---------|---------|
| pending | scheduled | 调度器取出任务 |
| pending | cancelled | 用户取消 |
| scheduled | running | Pod 启动成功 |
| scheduled | failed | Pod 创建失败 |
| running | paused | 用户暂停 |
| running | waiting_approval | 审批断点 (v1.1+) |
| running | succeeded | 任务完成 |
| running | failed | 任务失败 |
| paused | running | 用户恢复 |
| paused | cancelled | 用户取消 |
| failed | retrying | 自动重试 |
| retrying | running | 重试开始 |
```

### 5.2 错误处理规范

**位置**：TRD 7.3 错误码

**问题描述**：
定义了错误码，但缺少错误处理策略。

**建议补充**：

```markdown
### 7.6 错误处理策略

#### 重试策略
| 错误类型 | 重试 | 最大次数 | 退避 |
|---------|------|---------|------|
| 网络超时 | 是 | 3 | 指数退避 |
| 限流 (429) | 是 | 5 | 固定+抖动 |
| 校验失败 | 否 | - | - |

#### 幂等性
- POST 操作支持幂等键
- 幂等键 Redis TTL: 24h

#### 熔断器
- DB: 10s 内 5 次失败 → 熔断 30s
- 外部 API: 10s 内 3 次失败 → 熔断 60s
```

### 5.3 配置管理设计

**位置**：TRD 4 控制面服务设计

**问题描述**：
项目结构有 `config/config.go`，但缺少配置管理设计说明。

**建议补充**：

```markdown
### 4.5 配置管理

#### 配置来源优先级
1. 环境变量（最高）
2. K8s ConfigMap/Secret
3. 配置文件
4. 默认值（最低）

#### 敏感配置
- 数据库密码、API 密钥仅从环境变量读取
- 支持从 K8s Secret 挂载
```

### 5.4 软删除支持

**位置**：TRD 6.2 表结构

**建议**：
关键表（tasks, templates, users）添加 `deleted_at` 字段支持软删除，便于审计追踪。

### 5.5 API 限流规范

**位置**：TRD 7 API 设计

**建议补充**：

```markdown
### 7.7 API 限流
```
